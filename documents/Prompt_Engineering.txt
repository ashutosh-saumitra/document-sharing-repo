Prompt Engineering:

Goal: Clearly state what you want Copilot to do.

Context: Provide any relevant background, domain, audience, or constraints.

Source: Mention any documents, text, code, or examples Copilot should use.

Expectations: Describe exactly how you want the output to be structured or formatted.

👉 Prompting lesson: Always mention:

Why you’re doing it (learning, prod, POC)
Who it’s for (you, manager, senior architect, etc.)

👉 Prompting lesson: When you want step-by-step collaboration, explicitly describe:

The shape of each response (e.g., summary + options)
How you will respond (choose an option, ask for detail, etc.)

6️⃣ How to keep improving your prompt-writing technique
Here are a few advanced tips, tailored to your engineering background:


Think of prompts as “API contracts” for Copilot

Goal = Endpoint behavior
Context = Request metadata
Source = Payload
Expectations = Response schema



Separate “must-have” vs “nice-to-have”

Use wording like: “Must include…” and “If space allows, also include…”
This helps the model prioritize when output length is constrained.



Add constraints when needed
Examples:

“Keep explanations non-verbose; prefer concise bullet points.”
“Avoid framework-specific configuration details for now; focus only on design.”



Iterate on the prompt using the output

After first answer, refine: “Now redo the previous answer but also include X and skip Y.”
This is like agile refinement of API contracts.

Final Prompt:-
Goal:
Help me design a small RESTful API in Spring Boot from scratch, focusing on architecture, layers, and naming conventions before moving to any implementation details.
Context:
I want to create a simple Spring Boot microservice as a learning exercise. The service will manage IRIS employee details and expose CRUD operations over HTTP. I want to understand the design step-by-step so I can implement it myself later. There is no external audience; this is purely for my own learning and practice.
Source:
Use only the information provided in this prompt. If you need to make reasonable assumptions (e.g., naming, standard practices), follow common Spring Boot and REST best practices.
Expectations:


Architecture & Layers

Propose a standard Spring Boot layered architecture for this microservice (e.g., controller, service, repository, model/entity, exception handling package).
Explain the responsibility of each layer in simple terms.



Domain & Database

The service must use an in-memory H2 database.
Design a database table (and corresponding JPA entity) for storing IRIS Employee details.
Suggest typical fields for an employee (e.g., id, name, email, department, status, etc.) and explain each briefly.



API Design

Design RESTful endpoints that support GET, POST, PUT, PATCH, DELETE for the employee resource.
Provide correct naming conventions for:

Base URL path (e.g., /api/v1/employees)
Each endpoint (method + path) and its purpose


Briefly explain when to use PUT vs PATCH for this API.



Exception Handling

Describe how to structure global exception handling in this project (e.g., @ControllerAdvice, custom exception classes).
Give example exception types (e.g., EmployeeNotFoundException, validation errors) and how they should be returned as HTTP responses (status codes + response body structure).



Class & Package Design

Suggest package structure (e.g., controller, service, repository, model, dto, exception, config).
Propose class names for key components (controllers, services, repositories, entities, DTOs, exceptions).



Step-by-step Learning Flow

First: Explain the overall design and architecture in a step-by-step narrative (no code yet).
Then: Summarize the design as bullet points that list:

Main packages
Classes to create
Endpoints with HTTP method + path


At the end of your response, show numbered options (2–4 items) for what I can ask you next, for example:

“Generate sample class skeletons for the main components”
“Add validation rules and error response structure”
“Walk me through how a POST request flows through the layers”


I will select the next option/step so we can continue interactively.
